# CargoFlow: 3D Graph-Based Cargo Logistics Optimization System

## 1. Project Overview

CargoFlow is a graph-based cargo logistics optimization system designed to model cargo movement across a transportation network and help users analyze the most efficient routes, detect bottlenecks, and visualize the network in an interactive 3D dashboard. The system is inspired by CargoShield, but scaled down to a student project level with a strong DAA focus, a polished frontend, and a dataset-driven graph model. [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/attachments/70487251/8423962d-0681-475e-b87d-fd17d26b0504/CargoShield_Project_Report.pdf)

The core idea is to represent cargo-related entities such as ports, warehouses, terminals, cranes, gates, customs points, trucks, and depots as graph nodes, and represent movement or dependency relationships between them as weighted edges. The system then applies graph algorithms such as Dijkstra’s algorithm for shortest route optimization, BFS/DFS for traversal and reachability analysis, and Prim’s or Kruskal’s algorithm for minimum spanning network design. [codecademy](https://www.codecademy.com/article/dijkstras-shortest-path-algorithm)

The frontend will be built in React and will use a 3D graph visualization approach using React Three Fiber or a force-directed graph component so the dashboard looks visually impressive and interactive. React Three Fiber is a React renderer for Three.js, and the force-graph ecosystem supports 2D, 3D, VR, and AR style graph visualizations, which makes it suitable for an immersive graph dashboard. [github](https://github.com/vasturiano/react-force-graph)

## 2. Problem Statement

Cargo movement in logistics systems is often inefficient because route selection, network planning, and congestion handling are done using static rules or manual judgement. In a real cargo network, delays in one part of the system can propagate to other parts, causing longer travel times, congestion at hubs, and suboptimal resource use. The project addresses the following issues:
- Finding the shortest and most efficient cargo route between two nodes.
- Identifying whether some parts of the network are disconnected or poorly connected.
- Designing a cost-efficient infrastructure network.
- Visualizing the cargo graph clearly in 3D so that the results are easy to understand.

## 5. Algorithms Used

The project will use three major graph algorithms.

### 5.1 Dijkstra’s Algorithm
Dijkstra’s algorithm will be used to find the minimum-cost or minimum-time route between a source and destination node in the cargo logistics graph. It is suitable because the graph has non-negative weights, and the system wants the best path based on distance, time, or travel cost. [brilliant](https://brilliant.org/wiki/dijkstras-short-path-finder/)

### 5.2 BFS / DFS
BFS or DFS will be used for connectivity analysis, node reachability, and cluster traversal. This can help identify which cargo points are affected if a node becomes unavailable or which nearby nodes belong to the same operational zone.

### 5.3 Prim’s or Kruskal’s Algorithm
Prim’s or Kruskal’s algorithm will be used to compute the minimum spanning tree of the cargo network. This is useful to show the most cost-efficient way to connect hubs or design a reduced-cost transport structure. [youtube](https://www.youtube.com/watch?v=Uj47dxYPow8)

## 6. System Architecture

The project architecture will have four major layers.

### 6.1 Data Layer
This layer stores the graph dataset in CSV or JSON format. It contains node and edge information and can be loaded into memory when the app starts.

### 6.2 Algorithm Layer
This layer contains the implementations of:
- Dijkstra
- BFS/DFS
- Prim or Kruskal

It receives graph data from the data layer and returns path details, traversal results, or spanning tree results.

### 6.3 Backend Layer
This layer exposes the data and algorithm results through API endpoints. It can be implemented using Node.js/Express or Python FastAPI. The backend is responsible for:
- loading the graph,
- running selected algorithms,
- returning results to the frontend,
- handling scenario changes such as blocking a node or increasing an edge weight.

### 6.4 Frontend Layer
This is the React-based dashboard. It will render:
- a 3D graph scene,
- a metrics panel,
- route output cards,
- a node details drawer,
- algorithm comparison charts.

React Three Fiber is a strong fit for the 3D view because it allows declarative Three.js scenes in React. A force-directed graph component can also help produce a visually compelling network layout. [vasturiano.github](https://vasturiano.github.io/react-force-graph/)

## 9. API Design

The backend can expose the following endpoints:

- `GET /graph`  
  Returns full graph nodes and edges.

- `GET /node/{id}`  
  Returns details for one node.

- `POST /algorithm/dijkstra`  
  Runs shortest path between source and destination.

- `POST /algorithm/bfs`  
  Runs BFS traversal from a selected source.

- `POST /algorithm/dfs`  
  Runs DFS traversal from a selected source.

- `POST /algorithm/mst`  
  Runs Prim or Kruskal and returns the minimum spanning tree.

- `POST /scenario/block-node`  
  Simulates a node failure or blockage.

- `POST /scenario/update-edge`  
  Simulates congestion or delay on one edge.

- `GET /metrics`  
  Returns summary stats and computed outputs.

## 13. Complexity Analysis

### Dijkstra
- Time complexity: \(O(E \log V)\) with a priority queue.
- Space complexity: \(O(V + E)\).

### BFS / DFS
- Time complexity: \(O(V + E)\).
- Space complexity: \(O(V)\).

### Prim’s algorithm
- Time complexity: \(O(E \log V)\) with a heap.
- Space complexity: \(O(V + E)\).

### Kruskal’s algorithm
- Time complexity: \(O(E \log E)\).
- Space complexity: \(O(V + E)\).

These are standard DAA-level complexities and easy to explain in viva. [brilliant](https://brilliant.org/wiki/dijkstras-short-path-finder/)
