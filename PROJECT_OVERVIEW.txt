CargoFlow / CargoShield DAA Project Overview

This document explains the full structure of the project: system architecture, technology stack, file relationships, backend and frontend flow, data model, algorithms, and how the pieces connect.

1. Project Summary

This repository contains a two-part logistics visualization and optimization system:
- A FastAPI backend that loads maritime graph data, exposes REST endpoints, and runs routing/scenario logic.
- A React + Vite frontend that displays the cargo network in a 3D globe view and lets users run pathfinding and scenario simulations.

The main purpose of the app is to analyze a cargo shipping network using graph algorithms, especially Dijkstra's shortest path, and visualize the results interactively.

2. High-Level Architecture

The project is split into these major layers:
1. Data Layer
   - JSON files store the graph nodes and edges.
   - Scenario state is kept in memory in the backend.
2. Backend Layer
   - FastAPI serves graph data, metrics, node details, and algorithm outputs.
   - Python modules load the graph and compute shortest paths.
3. Frontend Layer
   - React components render the dashboard layout.
   - A global Zustand store manages application state.
   - Axios calls the backend API through a Vite proxy.
4. Visualization Layer
   - Three.js renders the 3D Earth globe.
   - Nodes, arcs, and path particles are drawn in the browser.

5. Backend Architecture

Entry Point:
backend/main.py creates the FastAPI app and exposes the REST API.

It imports:
- load_graph, get_node_map, compute_metrics from graph.py
- dijkstra from algorithms/dijkstra.py
- scenario helpers from algorithms/scenario.py

Backend Responsibilities:
- loading node and edge data
- applying scenario changes
- building the adjacency list
- computing graph summaries
- running shortest path queries
- serving node details

API Endpoints:
- GET /graph
  - Returns nodes, edges, and current scenario state
- GET /node/{node_id}
  - Returns one node
- GET /metrics
  - Returns graph summary metrics
- POST /algorithm/dijkstra
  - Runs shortest path on the current graph
- POST /scenario/block-node
  - Blocks a node
- POST /scenario/unblock-node
  - Restores a node
- POST /scenario/update-edge
  - Changes an edge congestion weight
- POST /scenario/reset
  - Resets all scenario changes
- GET /scenario
  - Returns current scenario state
- GET /
  - Health / root response

Data Loading Flow:
backend/graph.py loads:
- backend/data/nodes.json
- backend/data/edges.json

It then:
1. removes blocked nodes
2. removes edges connected to blocked nodes
3. applies congestion overrides
4. builds an adjacency list

The adjacency list is what the algorithm consumes.

Scenario State:
backend/algorithms/scenario.py keeps state in memory:
- blocked nodes
- congested edges

This state is not stored in a database. It resets when the backend restarts.

Dijkstra Implementation:
backend/algorithms/dijkstra.py:
- uses heapq as a priority queue
- supports three modes:
  - cost
  - distance
  - travel_time
- computes the shortest route based on the selected weight
- reconstructs:
  - path node IDs
  - edge IDs
  - total cost
  - total distance
  - total time

Important detail:
- the implementation multiplies the selected base weight by congestion_weight
- this makes congestion affect all three optimization modes in a similar way

That design choice is one reason the same route can appear across modes if the data is strongly correlated.

8. Data Model

Node Data
Each node contains fields like:
- node_id
- node_name
- node_type
- capacity
- congestion_level
- processing_time
- status
- lat
- lng

Edge Data
Each edge contains fields like:
- edge_id
- source
- target
- distance
- travel_time
- cost
- capacity
- congestion_weight
- edge_status

Graph Characteristics
The graph is loaded as an undirected network in the current implementation:
- each edge is added to the adjacency list in both directions

This is important because it simplifies routing, but it also reduces directional realism.

9. How the Files Connect

Backend Flow
1. main.py receives a request
2. It calls get_scenario_state() from scenario.py
3. It passes scenario state into load_graph() in graph.py
4. graph.py loads JSON data and builds nodes, edges, adjacency list
5. For routing, main.py calls dijkstra() in algorithms/dijkstra.py
6. The result is enriched with node metadata and returned as JSON

Frontend Flow
1. main.jsx mounts App.jsx
2. App.jsx calls fetchGraph() and fetchMetrics() on mount
3. useStore.js calls the Axios API client
4. Axios sends requests to /api/*
5. Vite proxies the request to the FastAPI server
6. The backend responds with JSON
7. Zustand stores the data
8. Components re-render using the updated store values

Visualization Flow
1. Graph3D.jsx receives nodes and edges from the store
2. It converts lat/lng into 3D positions
3. It draws nodes on the globe
4. It draws arcs for shipping lanes
5. It highlights the Dijkstra path if one exists
6. It animates particles along selected arcs

10. Algorithm Notes

Dijkstra
The project currently focuses on Dijkstra shortest path routing.

Strengths:
- correct shortest-path structure
- simple and fast for weighted graphs
- works well for the current graph size

Limitations in the current model:
- cost, distance, and time are highly correlated in the dataset
- congestion affects all modes similarly
- node-level delays are not part of the route cost
- the graph is undirected and therefore less realistic for maritime routing

What the README Mentions
The README also mentions:
- BFS / DFS traversal
- Prim's / Kruskal's MST

But in the current visible codebase, Dijkstra is the only clearly implemented algorithm.

11. Current Runtime Behavior

The app expects:
- backend running on http://localhost:8000
- frontend running on http://localhost:5173

The frontend fetches data immediately on load.

If the backend is not running:
- graph data will not load
- metrics will not load
- the UI may appear stuck in loading or empty states

12. Important Design Observations

Why the same route can appear for all modes
The current data and routing model cause route similarity because:
- distance, time, and cost are strongly correlated
- congestion multiplies all modes similarly
- there are no node costs
- there is little route asymmetry

So the algorithm is doing what it was designed to do, but the model is not expressive enough to produce very different winners across modes.

Why the explainability UI was added
The result panel now gives the user:
- the selected optimization mode
- a textual explanation
- average per-hop metrics
- a note about correlation in the dataset

This helps interpret the result without changing the backend algorithm.

15. Short Version

This project is a maritime cargo logistics dashboard with:
- a FastAPI backend
- a React/Vite frontend
- a Three.js 3D Earth visualization
- graph data stored in JSON
- Dijkstra routing
- scenario simulation for blocked nodes and congestion
- Zustand state management
- Axios-based backend communication

The backend owns the graph and algorithms.
The frontend owns the visual dashboard and user interaction.
